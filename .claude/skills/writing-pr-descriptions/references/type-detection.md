# Type Detection

Determine conventional-commit type from a cc-port PR diff with confidence-based analysis.

## Decision Tree

Apply in priority order. Stop at the first match.

1. Reverting commit? → `revert`
2. Only docs (`*.md`, `docs/**`, `AGENTS.md`, `CLAUDE.md`)? → `docs`
3. Only formatting / whitespace / `gofmt` output? → `style`
4. Only test files (`*_test.go`, `testdata/**`)? → `test`
5. Only build / packaging (`.goreleaser.yml`, `Makefile`)? → `build`
6. Only dependencies (`go.mod`, `go.sum`)? → `deps`
7. Only CI configs (`.github/workflows/**`, `.github/actions/**`)? → `ci`
8. New CLI subcommand, new flag, or new exported behaviour added? → `feat`
9. Fixes broken behaviour in an existing module? → `fix`
10. Measurable performance change? → `perf`
11. Code restructuring without observable behaviour change? → `refactor`
12. Otherwise → `chore`

## Project-Specific Type Guidance

| Change | Type | Notes |
|---|---|---|
| New `cc-port` subcommand under `cmd/cc-port` | `feat` | Scope = subcommand name (e.g. `move`, `export`, `import`) |
| New flag on an existing subcommand | `feat` | Scope = subcommand name |
| New exported function or type in `internal/<mod>` used by another module | `feat` | Scope = module name |
| Renamed or removed exported symbol | `refactor` | Often breaking (`!`); see Breaking-Change Detection |
| Bug-fix in a single module's behaviour | `fix` | Scope = module name |
| Cap-guard tightening in `internal/importer` (no behaviour change for valid input) | `fix` | Scope = `import` |
| Lock contract or session-keyed registry change | `refactor` | Often breaking (`!`); coordinate with `internal/lock`, `internal/claude` |
| Manifest XML schema change | `refactor` or `feat` | Breaking when older archives stop loading |
| `.goreleaser.yml` packaging change | `build` | Scope omitted unless tied to one platform |
| Workflow file added or modified | `ci` | Scope omitted for repo-wide workflows |
| `go.mod` / `go.sum` only | `deps` | Author often `renovate[bot]` / `dependabot[bot]` |
| Module README, AGENTS.md, or `docs/architecture.md` only | `docs` | Scope = module name when single module; omit for cross-cutting |
| `.golangci.yml`, `.editorconfig`, root config only | `chore` | Scope omitted |

## Scope Rules

Scope is the cc-port module from the PR's file classification.

- HIGH confidence: all changes under one of `cmd/cc-port/**`, `internal/<mod>/**` → scope = subcommand or module name.
- MEDIUM confidence: changes span one module and its direct dependencies → scope = the owning module.
- LOW confidence: three or more unrelated modules touched, no shared theme → omit scope and ask the user via `AskUserQuestion`.

Omit scope when:

- Type is `docs` and changes are repo-wide (`README.md`, `AGENTS.md`, `DEVELOPMENT.md`).
- Type is `ci` and only generic CI configs change.
- Type is `build` and the change isn't tied to one platform.
- Cross-cutting refactor with no single module owner.

## Confidence Levels

**HIGH**: One type clearly dominates.

- Only `_test.go` files → `test`.
- Only `.github/workflows/**` → `ci`.
- New file in `cmd/cc-port` registering a subcommand → `feat`.

**MEDIUM**: Primary type clear with incidental supporting changes.

- New flag in `cmd/cc-port` plus an updated module README → `feat` (README is incidental).
- Bug-fix in `internal/rewrite` plus a regression test → `fix`.

**LOW**: Multiple types equally valid. Use `AskUserQuestion`.

- New feature plus an unrelated bug fix.
- `refactor` and `feat` both arguable (e.g. extracted helper that also adds capability).

## Breaking-Change Detection

Mark as breaking (`!` after the type in the title; `BREAKING CHANGE:` block in the body) when any apply:

- Removed or renamed CLI subcommand, flag, or short alias.
- Removed or renamed exported Go symbol that another module references.
- Manifest XML schema change that breaks archives generated by older cc-port.
- Session-keyed registry change (see `internal/claude/README.md`).
- Lock-file contract change (see `internal/lock/README.md`).
- Import cap-guard change that rejects archives previously accepted.

NOT breaking:

- New CLI subcommand or flag added (additive).
- New exported symbol added.
- New optional field on the manifest XML schema.
- New session-keyed group entry that older state doesn't reference.
- Internal helper extraction with no exported-surface change.
